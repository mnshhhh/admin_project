# 高校固定资产全生命周期管理系统需求说明书 (requirements.md)

## 1. 引言

### 1.1 编写目的
本文档旨在定义“高校固定资产全生命周期管理系统”的完整功能与非功能需求。系统基于 **RBAC（基于角色的访问控制）** 模型，采用 **单一系统、响应式设计** 架构，旨在通过细粒度的**功能权限点**与**数据权限**控制，实现从“采购-入库-使用-维护-报废”的全流程闭环管理。

### 1.2 核心理念
*   **单一系统**：不再物理分割 PC 端与移动端系统，一套代码，多端适配。
*   **权限驱动**：所有功能入口（菜单、按钮、操作）均由权限点（如 `asset:add`, `audit:approve`）动态控制。
*   **数据隔离**：基于角色的数据范围（Data Scope），实现“校级-院级-个人”三级数据自动隔离。

---

## 2. 角色与权限模型 (RBAC + Data Scope)

系统不预设固定的“端”入口，而是根据用户拥有的**权限点**和**数据范围**动态渲染界面。

| 角色示例 | 数据范围 (Data Scope) | 核心功能权限点 (Permission Points) | 典型业务场景 |
| :--- | :--- | :--- | :--- |
| **普通师生** | **仅本人** (Self) | `asset:view` (查看本人), `asset:apply` (借用/报修), `user:profile` (个人中心) | 手机端查看名下资产、扫码报修、修改密码。 |
| **部门管理员** | **本部门及下级** (Dept) | `asset:manage` (部门资产增删改), `audit:approve` (初审), `dept:user:view` (查看本部门人员) | PC端批量入库、iPad端盘点扫码、审批本院借用申请。 |
| **校级管理员** | **全校数据** (All) | `sys:manage` (系统配置), `asset:audit:final` (终审), `report:view` (全校报表) | 维护组织架构、定义折旧规则、全局数据决策。 |
| **维修人员** | **工单相关** (Task) | `task:view` (查看工单), `task:complete` (完成维修) | 手机端接单、更新维修进度。 |

系统采用 **"模块:资源:操作"** 三段式命名规范统一定义所有权限点，前后端共用同一份权限标识。

| 模块 | 权限点标识 | 说明 | 适用角色示例 |
| :--- | :--- | :--- | :--- |
| **资产** | `asset:list` | 查看资产列表 | 全体用户 |

| | `asset:edit` | 修改资产基础信息 | 部门管理员、校级管理员 |
| | `asset:remove` | 删除资产 | 校级管理员 |
| | `asset:scan` | 扫码操作入口 | 全体用户（设备支持时展示）|
| | `asset:import` | Excel 批量导入 | 部门管理员、校级管理员 |
| | `asset:export` | 数据导出 | 管理员角色 |
| | `asset:borrow` | 发起借用申请 | 教职工、学生 |
| | `asset:return` | 归还操作 | 教职工、学生 |
| | `asset:allocate` | 跨部门调拨 | 部门管理员（发起）、校级管理员（终审）|
| | `asset:check` | 发起/执行盘点 | 管理员 |
| **维修** | `repair:add` | 提交报修申请 | 全体用户 |
| | `repair:manage` | 接收/处理维修工单 | 维修员、管理员 |
| **审批** | `audit:approve` | 审批各类申请 | 部门管理员、校级管理员 |
| | `audit:view` | 查看审批历史 | 全体用户（查看本人单据）|
| **系统** | `sys:user:list` | 查看用户列表 | 管理员 |
| | `sys:user:add` | 新增用户 | 校级管理员 |
| | `sys:user:edit` | 编辑用户信息 | 管理员 |
| | `sys:user:remove` | 禁用/删除用户 | 校级管理员 |

| | `report:depreciation` | 折旧财务报表 | 校级管理员 |

## 4. 功能需求详述 (按功能模块划分)

所有模块均支持响应式布局，功能可用性由 `perms` 权限标识决定。

### 4.1 资产核心管理模块 (Asset Core)
*   **资产入库 (`asset:add`)**
    *   支持单条录入与 Excel 批量导入。
    *   **自动赋码**：入库即生成唯一资产编码及二维码。
*   **资产查询与检索 (`asset:view`)**
    *   **全局检索**：支持按名称、编号、分类模糊搜索。
    *   **权限过滤**：普通用户只能搜到“可借用”或“名下”的资产；管理员可搜到管辖范围内的所有资产。
*   **资产卡片与详情**
    *   展示资产全貌（基础信息、流转记录、维保记录）。
    *   **动态操作栏**：根据权限显示“编辑”、“报修”、“报废”等按钮。

### 4.2 流程审批与流转模块 (Workflow)

*   **借用/领用申请 (`asset:borrow`)**

    *   师生扫码或检索发起申请，填写用途与归还时间。

    *   系统校验资产状态（需为“闲置”）。

*   **多级审批中心 (`audit:approve`)**

    *   **待办任务**：集中展示需要当前用户审批的单据（采购、借用、报废）。

    *   **移动审批**：支持手机端快速点击“通过”或“驳回”。

*   **跨部门调拨 (`asset:allocate`)**

    *   发起调拨 -> 调出方确认 -> 调入方确认 -> 数据归属权自动变更。



### 4.3 运维与盘点模块 (Operations)

*   **故障报修 (`repair:add`)**

    *   支持扫码快速报修，自动关联资产信息，上传故障图片。

*   **维修工单池 (`repair:manage`)**

    *   维修员抢单或指派模式，记录维修费用与耗时。

*   **数字化盘点 (`asset:check`)**

    *   **盘点任务**：管理员创建盘点任务（按部门/位置）。

    *   **扫码作业**：利用移动端摄像头连续扫码，系统实时比对“账面”与“实物”。

    *   **盘点报告**：自动生成盘盈盘亏报表。



### 4.4 系统治理模块 (System Governance)

*   **组织架构管理 (`sys:dept:manage`)**

    *   维护“学校-学院-系”树状结构。

*   **用户与角色管理 (`sys:user:manage`, `sys:role:manage`)**

    *   **权限分配**：为角色勾选功能菜单与操作按钮（权限点）。

    *   **数据权限配置**：指定角色的数据可见范围。

*   **审计日志 (`sys:log:view`)**

    *   记录所有敏感操作（谁、在什么时候、改了什么）。

## 5. 关键系统页面与功能映射



### 5.1 响应式工作台 (Dashboard)

*   **核心功能**：根据权限展示指标。管理员看到全校资产饼图/折旧报表；普通用户看到“扫一扫”与“我的借用”。

*   **权限点**：`report:asset`, `asset:scan`, `audit:view`。



### 5.2 资产管理控制台 (Asset Console)

*   **核心功能**：统一的资产列表，支持权限按钮（新增、导入、导出、批量修改）。

*   **权限点**：`asset:list`, `asset:add`, `asset:import`, `asset:export`。



### 5.3 审批中心 (Approval Hub)

*   **核心功能**：跨单据类型的统一审批列表。

*   **权限点**：`audit:approve`。



### 5.4 系统配置中心 (Admin Settings)

*   **核心功能**：账号维护、角色权限点勾选、部门树维护。

*   **权限点**：`sys:user:manage`, `sys:role:manage`, `sys:dept:manage`。

## 6. 非功能需求



### 6.1 权限控制要求
*   **前端控制**：菜单栏、操作按钮、页面元素需根据后端返回的权限列表（如 `['asset:add', 'user:view']`）自动隐藏或禁用。
*   **后端控制**：所有接口必须具备鉴权注解（如 `@PreAuthorize("hasPermission('asset:add')")`），严防越权访问。
*   **数据隔离**：SQL 查询需自动注入部门过滤条件，防止低权限管理员查看其他部门数据。

### 4.2 交互体验要求
*   **响应式布局**：核心页面（表格、表单、详情页）需适配 Desktop (1920px), Tablet (768px), Mobile (375px)。
*   **渐进式增强**：在支持摄像头的设备上自动启用“扫一扫”功能，不支持的设备降级为“手动输入编码”。

### 4.3 性能要求
*   **权限校验延迟**：< 50ms。
*   **并发支持**：支持全校教职工同时在线进行年底盘点。
